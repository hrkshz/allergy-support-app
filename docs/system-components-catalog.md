# システム構成コンポーネント一覧

---

## この資料の役割

この資料は、[`portfolio-system-overview.md`](./portfolio-system-overview.md) を読んだときに、
**「登場人物が多くて、どれが何なのか分からなくなる」** を防ぐための補助資料です。

目的は次の3つです。

- このシステムに出てくる要素を、一覧で見渡せるようにする
- それぞれが `サーバなのか / DBなのか / 外部APIなのか` を判別しやすくする
- 初心者でも「何のためにいる部品か」がつかめるようにする

このアプリは、1台の大きなサーバですべてを処理する構成ではありません。  
**画面、サーバ処理、外部サービス、保存先、ルール** を役割ごとに分けて組み合わせる構成です。

## 1. 分類つきコンポーネント一覧

| コンポーネント名 | 分類 | 何者か（初心者向け） | このシステムでの役割 |
|---|---|---|---|
| **ユーザー** | 人 | アプリを使う本人 | ログインし、食品表示を撮影し、結果や履歴を確認する |
| **フロント全体** | 画面 / UI | ブラウザやスマホで開く、ユーザーが直接見るアプリ画面のまとまり | `Vercel` から静的ファイルとして配信され、ブラウザ上で動くUIとして入力、撮影、結果表示、履歴確認の入口になる |
| **Vercel** | サーバ処理 | 今回のフロント配信先 | React + TypeScript をビルドした HTML / CSS / JavaScript を置き、HTTPS で配信する |
| **静的ファイル置き場** | ストレージ | フロントの HTML / CSS / JavaScript を置いておく場所 | 今回の前提では `Vercel` 上に配置されるフロント成果物の保管先にあたる |
| **静的ファイル配信** | サーバ処理 | 置かれた静的ファイルをブラウザへ届ける仕組み | 今回の前提では `Vercel` がユーザーのブラウザにフロントアプリを配る |
| **ブラウザ上のフロントロジック** | 画面 / UI | ユーザーの端末で実際に動くフロントのコード | 画面表示、入力処理、画面遷移、Edge Functions の呼び出しを行う |
| **ログイン画面** | 画面 / UI | アプリを使い始める最初の画面 | ユーザーが認証し、自分の設定と履歴に入れるようにする |
| **プロフィール登録画面** | 画面 / UI | 自分の条件を最初に登録する画面 | 登録アレルゲンや重症度を設定し、以後の判定に使う |
| **撮影画面** | 画面 / UI | 食品パッケージを撮るための画面 | 原材料表示の画像を受け取り、判定開始のきっかけを作る |
| **結果画面** | 画面 / UI | 判定の答えを見る画面 | `検出なし / 要確認 / 検出あり` と理由、注意文を表示する |
| **履歴画面** | 画面 / UI | 過去の確認結果を見返す画面 | 前に確認した商品の結果や理由を再確認できるようにする |
| **API / Edge Functions 全体** | サーバ処理 | フロントの代わりに裏側処理を受ける、軽いバックエンド処理 | Supabase Edge Functions として動き、AI呼び出し、ルール照合、保存、返却データの組み立てをまとめる |
| **判定実行API** | サーバ処理 | 判定を始めるための受付窓口 | 画像参照先、プロフィール、ルールを集めて判定処理を進める |
| **保存処理** | サーバ処理 | 判定後の情報を記録する裏側処理 | 判定結果、理由、撮影情報をDBへ保存する |
| **レスポンス整形処理** | サーバ処理 | 裏側の結果を画面向けに見やすく変える処理 | 判定結果、理由、補足、免責文言をUI向けの形にまとめる |
| **OpenAI API** | 外部AI API | OpenAIが提供するAI機能を呼び出す外部サービス | 画像や原材料表示を読み、危なそうな候補や理由の材料を返す |
| **Supabase Auth** | 認証サービス | ログイン機能をまとめて提供する外部サービス | 誰が使っているかを識別し、ユーザーごとのデータと結びつける |
| **Supabase Storage** | ストレージ | 画像などのファイルを置く保管庫 | 撮影画像そのものを保存し、あとで履歴から参照できるようにする |
| **Supabase Database** | データベース | 表形式で情報を保存するデータ置き場 | プロフィール、登録アレルゲン、履歴、画像参照先を保存する |
| **ユーザープロフィールデータ** | データ | ユーザーごとの基本設定データ | 誰の設定か、どんな条件で判定するかの土台になる |
| **登録アレルゲンデータ** | データ | 何に気をつけるべきかを表すデータ | その人にとって危険かどうかを判定するときの基準になる |
| **重症度データ** | データ | アレルギーの重さに関する設定データ | 将来的に注意の出し方や優先度を変える判断材料になる |
| **判定履歴データ** | データ | 過去の判定結果をためておく記録 | 後から同じ商品や以前の判断理由を見返せるようにする |
| **画像参照データ** | データ | 保存済み画像の場所を示すデータ | DBの判定履歴とStorage上の画像をつなぐ |
| **撮影画像ファイル** | ストレージ | ユーザーが撮った実際の画像ファイル | AI読解の入力になり、あとで履歴から見返す対象にもなる |
| **ルール辞書** | 設定ファイル / ルール | アプリ共通で使う判断ルールの集まり | 確定語、曖昧語、除外語などを持ち、AI結果の解釈に使う |
| **確定語リスト** | 設定ファイル / ルール | 見つかったら危険寄りに判断しやすい語の一覧 | `検出あり` に近い判断材料として使う |
| **曖昧語リスト** | 設定ファイル / ルール | 由来が断定しにくい語の一覧 | 無理に白黒つけず `要確認` に寄せる材料にする |
| **除外語リスト** | 設定ファイル / ルール | 文字が似ていても別物として扱う語の一覧 | 誤検出を減らし、単純な文字一致を補正する |
| **表示文言テンプレート** | 設定ファイル / ルール | 結果画面の言い回しの型 | 判定理由、補足、注意文を一定の分かりやすさで返す |
| **免責文言** | 設定ファイル / ルール | 「安全保証ではない」と伝えるための文言 | `検出なし` を過信させないようにする |
| **判定ロジック** | アプリ内ロジック | AI結果とルールを合わせて最終判定を作る考え方と処理 | `検出なし / 要確認 / 検出あり` を安全側で決める |
| **AI読解結果** | データ | OpenAI API から返ってくる読み取り結果 | 原材料候補、怪しい表記、理由の材料として判定ロジックに渡る |
| **総合判定結果** | データ | 最終的に画面へ返す判定データ | 結果画面でユーザーに見せる中心情報になる |

## 3. 初心者向けにざっくり一言でいうと

このシステムは、次のように役割分担されています。

| 役割 | 担当 |
|---|---|
| ユーザーが触る部分 | フロントと各画面 |
| 裏側で処理をまとめる部分 | API / Edge Functions |
| AIで画像や文字を読む部分 | OpenAI API |
| ログインを管理する部分 | Supabase Auth |
| 画像ファイルを置く部分 | Supabase Storage |
| 設定や履歴をためる部分 | Supabase Database |
| アプリ独自の判断ルールを持つ部分 | ルール辞書 |
| AI結果を安全側に整える部分 | 判定ロジック |

つまり、**1つの大きなサーバで全部やるのではなく、必要な役割ごとに部品を分けて組み合わせている**、という理解で大丈夫です。

### 補足: フロントサーバーはあるのか

この資料の前提では、**専用のフロントサーバーは置かない**。

- フロントは `Vercel` に静的ファイルとして置かれ、配信され、ブラウザで動く
- 裏側のサーバー的な処理は `Supabase Edge Functions` で受ける

つまり、**フロントサーバーは不要だが、サーバー処理は必要**という整理になる。

AWS のイメージでたとえると、`Vercel` が `S3 + CloudFront` に近い役割をまとめて担い、ブラウザが `フロントロジックの実行場所` にあたる。

SSR、SEO重視、サーバー側での複雑な画面生成が必要になった場合は、将来的に別構成を検討する。
