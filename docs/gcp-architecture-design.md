# GCPアーキテクチャ設計 — パイプライン分離によるアレルゲン判定

---

## この資料の役割

この資料は、アレルギー支援アプリの**GCPを活用したアーキテクチャ設計**をまとめた資料です。

読めば次の5つがわかるようにしています。

- なぜGCPを選ぶのか
- パイプライン分離で何が変わるのか
- GCPのどのサービスをどう使うのか
- このスキルセットがなぜ市場価値を持つのか
- 将来どう拡張できるのか

現行構成（Vercel + Supabase + OpenAI API）の全体像は [`portfolio-system-overview.md`](./portfolio-system-overview.md) に、
判定ロジックの詳細は [`allergen-detection-logic.md`](./allergen-detection-logic.md) にまとめています。

---

## 1. 現行構成の整理 — 出発点を明確にする

現行構成では、OpenAI API が「画像読み取り＋原材料理解＋アレルゲン判定」を**一括で処理**している。

```
撮影 → OpenAI API（OCR + 理解 + 判定を一括） → 結果表示
```

これは MVP を速く作るには合理的な判断だった。
しかし、次の課題がある。

- **OCRミスなのかAI判定ミスなのか切り分けられない**
- **OCR精度だけ改善したい、ができない**（AI全体を変えるしかない）
- **毎回の判定に汎用LLMの料金がかかる**（OCR部分は専用サービスの方が安い）
- **中間データが見えない**ので、デバッグやテストがしにくい

---

## 2. パイプライン分離 — 何が変わるか

一括処理を5つのステップに分ける。

```
撮影 → OCR → 文字処理 → 整形 → AI判定 → 結果表示
```

### 各ステップの責務

| ステップ | 入力 | 出力 | やること |
|---|---|---|---|
| **撮影** | ユーザー操作 | 画像ファイル | 食品パッケージの原材料欄を撮る |
| **OCR** | 画像ファイル | 生テキスト | 画像から文字を抽出する（精度重視） |
| **文字処理** | 生テキスト | 正規化テキスト | ノイズ除去、誤認識修正、表記ゆれ統一 |
| **整形** | 正規化テキスト | 構造化データ | 原材料リスト・添加物・アレルゲン表示に分離 |
| **AI判定** | 構造化データ + プロファイル | 判定結果 | 安全/要確認/危険の判定と理由生成 |

### ここで大事なこと

分離すると、こうなる。

- OCR の精度が悪ければ、**OCR だけ差し替えられる**
- 文字処理のルールが足りなければ、**ルールだけ追加できる**
- AI判定のプロンプトを変えても、**前段には影響しない**
- 各ステップの出力を保存すれば、**問題の原因がすぐわかる**

つまり、**改善のサイクルが速くなる**。

## 3. なぜGCPか — 技術選定の根拠

GCP を選ぶ理由は「なんとなく」ではない。このアプリのパイプラインに対して、GCP が構造的に有利な点がある。

### 3.1 Cloud Vision API — 日本語食品ラベルに強い

OCR ステップにおいて、Cloud Vision API は次の特性を持つ。

- **日本語テキスト認識の精度が高い**。漢字・ひらがな・カタカナの混在、小さい文字、高密度の表示に対応
- **DOCUMENT_TEXT_DETECTION** モードでは、段落・行・単語の構造を保持したまま返すため、後続の文字処理が楽になる
- **食品ラベルのような自由配置テキスト**に対応できる（帳票専用ではない）
- 画像ファイルを Cloud Storage に置けば、API に URL を渡すだけで処理できる

### 3.2 Vertex AI (Gemini) — パイプラインとの統合が自然

AI判定ステップでは、Vertex AI (Gemini) が有利。

- GCP のサービスとしてネイティブに統合されている（認証、ネットワーク、ログがすべてGCP内で完結）
- **構造化された入力を受けて、構造化された出力を返す**パターンに適している
- OCR + 理解を一括でやる必要がないため、**判定と理由生成に集中させられる**
- Gemini の pricing は GPT-4o と比較して競争力がある

### 3.3 Firebase エコシステム — 周辺機能がそろっている

パイプライン以外の機能（認証、DB、Hosting）も、Firebase を使えばGCP内で一貫して管理できる。

- **Firebase Auth**: ソーシャルログイン、メール認証がすぐ使える。モバイルSDKも充実
- **Firestore**: リアルタイム同期、オフライン対応が内蔵。モバイル展開時に大きなメリット
- **Firebase Hosting**: CDN 統合、SSL 自動、`firebase deploy` 一発でデプロイ
- **Cloud Storage**: 撮影画像の保存先。Vision API との連携が直接的

### 3.4 IDP 領域への Google の投資

Google は **Document AI** というマネージドサービスを出している。
これは「画像から情報を抽出して構造化する」パイプラインそのもので、このアプリのアーキテクチャと思想が重なる。

つまり、Google はこの領域に本気で投資しており、Cloud Vision API や Vertex AI の改善が今後も続く可能性が高い。

## 4. GCPサービス構成 — 全体マッピング

### 現行構成との対比

| パイプライン | 現行（一括） | GCP（分離） | 変わること |
|---|---|---|---|
| 撮影 | React (カメラAPI) | 同左 | 変更なし |
| 画像保存 | Supabase Storage | Cloud Storage | Vision API と直接連携 |
| OCR | OpenAI Vision (一括) | **Cloud Vision API** | 専用OCRで精度向上、コスト削減 |
| 文字処理 | — (AIに依存) | **Cloud Functions** | ルールベースで確実に処理 |
| 整形 | — (AIに依存) | **Cloud Functions** | 構造化データを明示的に作る |
| AI判定 | OpenAI API (一括) | **Vertex AI (Gemini)** | 判定と理由生成に集中 |
| 認証 | Supabase Auth | Firebase Auth | モバイル対応が強化 |
| DB | Supabase (PostgreSQL) | Firestore | オフライン対応、リアルタイム同期 |
| フロント配信 | Vercel | Firebase Hosting | GCP内で一元管理 |
| API層 | Edge Functions | **Cloud Run** | コンテナベースで自由度向上 |

### サービス全体像

```
[ユーザー]
   │
   ▼
[Firebase Hosting] ── React + TypeScript (フロント配信)
   │
   ├── Firebase Auth (ログイン)
   │
   ├── Cloud Storage (撮影画像の保存)
   │       │
   │       ▼
   │   [Cloud Vision API] ── OCR (画像→生テキスト)
   │       │
   │       ▼
   │   [Cloud Functions] ── 文字処理 (ノイズ除去・誤認識修正)
   │       │
   │       ▼
   │   [Cloud Functions] ── 整形 (構造化データに変換)
   │       │
   │       ▼
   │   [Vertex AI / Gemini] ── AI判定 (安全/要確認/危険 + 理由生成)
   │
   ├── Cloud Run (API層 / オーケストレーション)
   │
   └── Firestore (プロファイル・履歴保存)
```

## 5. 各ステップの技術詳細

### 5.1 OCR — Cloud Vision API

**使うAPI**: `DOCUMENT_TEXT_DETECTION`（`TEXT_DETECTION` より構造保持が優れる）

**なぜ DOCUMENT_TEXT_DETECTION か:**
- 食品ラベルは段落構造を持つ（原材料名：の後に続く一覧）
- 行ごとの区切りを保持してくれるため、後続の文字処理でパースしやすい
- `TEXT_DETECTION` は単語単位で返すため、文脈がバラバラになりやすい

**日本語食品ラベルでの注意点:**
- 小さい文字（7pt以下）は撮影品質に依存する。ユーザーへのガイダンス（「近くで撮ってください」）が必要
- 反射やぼやけは前処理（画像の明るさ・コントラスト調整）で改善できる場合がある
- 縦書きと横書きの混在は Cloud Vision API が対応しているが、結果の並び順に注意

**レスポンスの構造:**

```
画像 → Cloud Vision API → {
  fullTextAnnotation: {
    text: "原材料名：小麦粉、砂糖、鶏卵、バター...",
    pages: [{ blocks: [{ paragraphs: [{ words: [...] }] }] }]
  }
}
```

`fullTextAnnotation.text` を取るのが最もシンプルだが、精度検証の際は `pages` の構造も確認する。

### 5.2 文字処理 — Cloud Functions

OCR の生出力には、次のようなノイズが含まれる。

| 問題 | 例 | 処理 |
|---|---|---|
| 余計なスペース | `鶏 卵` → `鶏卵` | 日本語文字間のスペース除去 |
| OCR誤認識 | `パター` → `バター` | よくある誤認識の辞書マッチ |
| 改行の混入 | `原材料名：\n小麦粉` → `原材料名：小麦粉` | 不要な改行の結合 |
| 記号のゆれ | `（` vs `(` | 全角・半角の統一 |
| 類似文字の混同 | `口(くち)` vs `ロ(ロ)` | コンテキストに基づく修正 |

**ここで大事なこと:**

この文字処理はルールベースで書く。AIに任せない。
理由は次の通り。

- **確定的な処理は確定的にやるべき**。「パター」は常に「バター」に直す。AIに聞く必要はない
- ルールベースなら**テストが書ける**。入力と出力が決まっているので、自動テストで品質を保証できる
- AIに渡すテキストが綺麗になることで、**AI判定の精度も上がる**

### 5.3 整形 — Cloud Functions

正規化されたテキストを、構造化データに変換する。

食品表示は JAS 法（食品表示法）に基づいてフォーマットがほぼ決まっている。

**入力:**
```
原材料名：小麦粉（国内製造）、砂糖、鶏卵、バター、えびエキス（えびを含む）／膨張剤、着色料（カラメル）
```

**出力:**
```json
{
  "raw_text": "原材料名：小麦粉（国内製造）、砂糖、鶏卵、バター、えびエキス（えびを含む）／膨張剤、着色料（カラメル）",
  "ingredients": [
    { "name": "小麦粉", "origin": "国内製造" },
    { "name": "砂糖" },
    { "name": "鶏卵" },
    { "name": "バター" },
    { "name": "えびエキス", "allergen_note": "えびを含む" }
  ],
  "additives": [
    { "name": "膨張剤" },
    { "name": "着色料", "detail": "カラメル" }
  ],
  "allergen_labels": ["えび"]
}
```

**パースのルール:**
- `／`（全角スラッシュ）または `/` で**原材料と添加物を分離**する（食品表示法の規定）
- `、`（読点）または `，` で**個別の原材料を分割**する
- `（）` 内の「〜を含む」は**アレルゲン表示**として抽出する
- `（）` 内の産地情報（「国内製造」等）は**原産地表示**として保持する

**ここで大事なこと:**

この整形もルールベースで書く。食品表示法のフォーマットは法律で決まっているため、パターンが限られている。
AIに任せると、法律に基づかない「それっぽい」解析をしてしまうリスクがある。

### 5.4 AI判定 — Vertex AI (Gemini)

ここまでのステップで、AIに渡すデータは**綺麗に構造化されている**。
AI の仕事は次の2つに絞られる。

1. **ユーザーのアレルゲンプロファイルと構造化データを照合**して、安全/要確認/危険を判定する
2. **判定理由をユーザーにわかりやすい日本語で説明**する

**AIに任せるべき理由:**
- 「乳化剤」に「乳」が含まれるが乳アレルギーとは無関係、のような**文脈判断**はルールだけでは網羅しきれない
- 「本品製造工場では〜を含む製品を生産しています」のような**コンタミネーション表示の解釈**は文脈理解が必要
- 判定理由の**自然な日本語での説明生成**はAIの得意分野

**プロンプト設計の方針:**
- 入力: 構造化された原材料データ + ユーザーのアレルゲンプロファイル
- 出力: JSON形式の判定結果（安全/要確認/危険、理由、該当成分）
- 指示: 「確信がない場合は必ず『要確認』を返すこと」（安全側に倒す）
- ルールベースの判定結果も入力に含め、AIはそれを補完・検証する役割

**ルールベースとAIの役割分担:**

| 判断の種類 | 担当 | 例 |
|---|---|---|
| 明確な一致 | ルールベース | 「えび」→ えびアレルギーに該当 |
| 代替表記 | ルールベース（辞書） | 「海老」「エビ」→ えび |
| 文脈依存 | AI | 「乳化剤」→ 乳アレルギーには無関係 |
| 曖昧表現 | AI | 「一部に〜を含む」の解釈 |
| コンタミ表示 | AI | 「同じ工場で〜」の判断 |
| 理由説明 | AI | ユーザー向けのわかりやすい文章生成 |

## 6. このスキルセットの市場価値

### 6.1 IDP（Intelligent Document Processing）は成長領域

このアプリで実装するパイプライン — 「画像から非構造化テキストを取り出し、処理して、構造化データにする」 — は、IDP（Intelligent Document Processing）と呼ばれる領域に属する。

IDP は、あらゆる業界で需要がある。

| 業界 | 応用例 |
|---|---|
| **金融** | 請求書・領収書の自動読み取り → 会計システム連携 |
| **医療** | 処方箋・検査結果の電子化 → 電子カルテ統合 |
| **物流** | 伝票・ラベルの読み取り → 在庫管理自動化 |
| **不動産** | 契約書・登記簿の解析 → データベース化 |
| **製造** | 検査報告書 → 品質管理データ |
| **小売/食品** | このアプリ。商品ラベル → 成分情報の構造化 |

### 6.2 「APIを叩ける」と「パイプラインを設計できる」の差

Cloud Vision API を呼べるだけでは差別化にならない。
このアプリで得られるのは、次のスキルの**組み合わせ**。

- OCR 生出力のクセを理解して**後処理を設計できる**
- ルールベース処理と AI 処理を**適切に使い分けられる**
- 精度が異なるステップを組み合わせて**全体の信頼性を設計できる**
- 各ステップを独立にテスト・改善する**パイプライン運用ができる**

これは「アレルギーアプリを作りました」ではなく、**「非構造化データを構造化するパイプラインを設計・実装できます」**というポータブルなスキルになる。

### 6.3 トレンドとの整合

- **マルチモーダルAI + OCR**: 従来のOCR単体から「OCR + LLMで意味理解」へ進化。このパイプラインはまさにその構成
- **企業のペーパーレス化**: 日本はまだ紙文化が残っており、この技術の需要が特に大きい
- **GCPの AI/ML ポジション**: AWS はインフラ全般が強いが、GCP は AI/ML 領域で差別化。この分野を学ぶなら GCP との相性が良い

### 6.4 キャリアパスとの接続

- **GCP 認定資格**: Cloud Digital Leader → Professional Cloud Architect への足がかり
- **求人キーワード**: AIエンジニア、MLOps、データパイプラインエンジニア、DXコンサルタント
- **日本市場での GCP**: データ/ML系企業やスタートアップで成長中。AWS の求人数には及ばないが、AI 特化で差別化できる

## 7. コスト試算 — MVP規模

MVP規模（~100 ユーザー、~1,000 判定/月）でのコスト概算。

| サービス | 無料枠 | MVP使用量 | 月額目安 |
|---|---|---|---|
| Firebase Hosting | 10GB/月 | 十分 | $0 |
| Firebase Auth | 50,000 MAU | 十分 | $0 |
| Cloud Storage | 5GB | 十分 | $0 |
| **Cloud Vision API** | 1,000 リクエスト/月 | ~1,000 | $0（枠内）〜$1.5 |
| **Vertex AI (Gemini)** | — | ~1,000 | $5〜15 |
| Cloud Functions | 200万回/月 | 十分 | $0 |
| Firestore | 1GiB, 50,000 読取/日 | 十分 | $0 |
| **合計** | | | **$5〜17/月** |

### ここで大事なこと

- 現行構成（OpenAI API 一括）は $10〜30/月程度。GCP分離構成は**同等かやや安い**
- パイプライン分離のメリットはコストだけではない。精度改善、デバッグ容易性、スキル獲得が本質
- 成長規模（10,000 ユーザー）になると、専用OCR + 軽量AI の分離構成がコスト面でも明確に有利になる

## 8. 将来拡張への道筋

GCP エコシステム内で、追加のサービスを導入するだけで拡張できる。

| 将来機能 | GCPサービス | 難易度 |
|---|---|---|
| バーコードスキャン | Cloud Vision API（barcode detection 内蔵） | 低 |
| オフライン対応 | Firestore offline sync | 中（設計が必要） |
| プッシュ通知 | Firebase Cloud Messaging (FCM) | 低 |
| 多言語対応 | Cloud Translation API | 低 |
| 音声読み上げ | Cloud Text-to-Speech | 低 |
| ネイティブアプリ | Flutter + Firebase | 中〜高 |
| アレルゲンDB API化 | Cloud Endpoints / Apigee | 中 |

### ここで大事なこと

これらの拡張は**パイプラインの設計を壊さない**。
バーコードスキャンは「撮影」ステップに入力を追加するだけ。多言語対応は「結果表示」ステップに翻訳を挟むだけ。

パイプラインを分離しておいたことで、**拡張時に既存の処理を変更しなくて済む**。

## 9. AWSとの簡潔比較

GCP を選ぶ理由を補強するために、AWS との差を簡潔に整理する。

| 観点 | GCP | AWS | このアプリにとって |
|---|---|---|---|
| **OCR** | Cloud Vision API | Textract | Textract は帳票・フォーム向き。食品ラベルのような自由配置テキストには Cloud Vision が向く |
| **認証** | Firebase Auth | Cognito | Cognito は高機能だが設定が複雑。Firebase Auth の方が学習コストが低い |
| **サーバーレス** | Cloud Run / Cloud Functions | Lambda | Cloud Run はコンテナベースでベンダーロックインが小さい。Lambda はエコシステムが広大 |
| **AI/ML** | Vertex AI (Gemini) | Bedrock (Claude, Titan) | Bedrock はマルチモデルが強み。Vertex AI は GCP との統合が自然 |
| **DB** | Firestore | DynamoDB | どちらも NoSQL。Firestore はオフライン同期が内蔵 |
| **モバイル** | Firebase（Flutter連携強い） | Amplify | Firebase の方がモバイル開発の実績・コミュニティが厚い |
| **市場シェア（日本）** | 成長中（AI/ML系で強い） | 最大（50%超） | AWS スキルの方が求人は多いが、AI特化なら GCP で差別化できる |

### 正直に言うと

- AWS の方が求人数は多い。インフラ全般をカバーしたいなら AWS
- しかし、このアプリは **AI/ML パイプライン**が中心。その領域では GCP が自然な選択
- AWS を「知らない」のではなく「このユースケースで GCP を選んだ理由を説明できる」ことが大事

## 10. この資料を1分で説明すると

> 現行構成の OpenAI API 一括処理を、「OCR → 文字処理 → 整形 → AI判定」に分離する。
> 各ステップを GCP のサービス（Cloud Vision API、Cloud Functions、Vertex AI）で実現し、
> Firebase（Auth、Firestore、Hosting）で周辺機能をそろえる。
>
> この構成は、OCR精度の向上、コスト最適化、デバッグ容易性をもたらすだけでなく、
> **「非構造化データを構造化するパイプラインを設計できる」**という
> IDP 領域で市場価値の高いスキルの獲得につながる。
